# Механизм тегирования данных

## 1. Структура тегов

### 1.1 Формат тегов

Используется формат ключ-значение: `namespace:key=value` или `namespace:value`

Примеры:
- `confidentiality:critical`
- `data_type:phi`
- `category:diagnosis`
- `gdpr:special_category=true`
- `medical_secret:true`

### 1.2 Основные пространства имён (namespaces)

#### 1.2.1 confidentiality (Уровень конфиденциальности)
**Описание:** Определяет уровень конфиденциальности данных

**Возможные значения:**
- `confidentiality:public` - Публичные данные
- `confidentiality:low` - Низкая конфиденциальность
- `confidentiality:medium` - Средняя конфиденциальность
- `confidentiality:high` - Высокая конфиденциальность
- `confidentiality:critical` - Критическая конфиденциальность

**Применение:**
- Определяет требования к шифрованию
- Определяет уровень аудита
- Определяет политики доступа

#### 1.2.2 data_type (Тип данных)
**Описание:** Категоризация данных по типу

**Возможные значения:**
- `data_type:pii` - Персональные данные
- `data_type:phi` - Медицинские данные
- `data_type:financial` - Финансовые данные
- `data_type:payment_card` - Данные платёжных карт
- `data_type:employee_data` - Данные сотрудников
- `data_type:contract` - Договорные данные
- `data_type:operational` - Операционные данные
- `data_type:analytical` - Аналитические данные

**Применение:**
- Определяет требования к защите по типу данных
- Применяет соответствующие политики (PCI DSS, 152-ФЗ)

#### 1.2.3 category (Категория данных)
**Описание:** Детальная категоризация данных

**Возможные значения для PII:**
- `category:name` - ФИО
- `category:birthdate` - Дата рождения
- `category:contact` - Контактные данные (телефон, email)
- `category:address` - Адрес
- `category:identity` - Идентификационные данные (паспорт, СНИЛС)

**Возможные значения для PHI:**
- `category:diagnosis` - Диагнозы
- `category:medical_history` - История болезни
- `category:lab_results` - Результаты анализов
- `category:prescriptions` - Назначения лекарств
- `category:insurance` - Страховые данные

**Возможные значения для финансовых данных:**
- `category:payment_amount` - Сумма оплаты
- `category:payment_method` - Способ оплаты
- `category:receipt` - Чеки

**Применение:**
- Детальная классификация для применения специфических политик
- Улучшение поиска и управления данными

#### 1.2.4 medical_secret (Врачебная тайна)
**Описание:** Данные, подпадающие под врачебную тайну

**Возможные значения:**
- `medical_secret:true` - Данные подпадают под врачебную тайну
- `medical_secret:false` - Данные не подпадают под врачебную тайну

**Применение:**
- Строгий контроль доступа
- Обязательное логирование всех обращений

#### 1.2.5 pci_dss (Требования PCI DSS)
**Описание:** Данные, подпадающие под требования PCI DSS

**Возможные значения:**
- `pci_dss:true` - Данные подпадают под PCI DSS
- `pci_dss:scope=cardholder_data` - Область действия PCI DSS

**Применение:**
- Применение требований PCI DSS
- Строгий контроль доступа

#### 1.2.6 data_owner (Владелец данных)
**Описание:** Владелец данных (отдел/роль)

**Возможные значения:**
- `data_owner:reception` - Ресепшен
- `data_owner:medical` - Медицинские специалисты
- `data_owner:accounting` - Бухгалтерия
- `data_owner:warehouse` - Склад
- `data_owner:it` - IT-отдел

**Применение:**
- Определение ответственности за данные
- Эскалация при нарушениях

#### 1.2.7 retention (Срок хранения)
**Описание:** Срок хранения данных

**Возможные значения:**
- `retention:period=365` - Срок хранения в днях
- `retention:policy=medical_records` - Политика хранения

## 3. Инструменты для тегирования

### Apache Atlas
**Описание:** Открытая платформа для управления метаданными и тегирования данных

**Возможности:**
- Классификация данных через теги
- Управление политиками безопасности
- Интеграция с Hadoop, Spark, Kafka
- Поиск данных по тегам
- Аудит доступа к тегированным данным

**Применение:**
- Централизованное управление тегами
- Автоматическое применение политик
- Поиск данных по тегам
